### 发版前准备(如果已配置请忽略)

- GitHub 仓库需要配置 `NPM_TOKEN`

```text
Settings -> Secrets and variables -> Actions -> Repository secrets -> NPM_TOKEN
```

- `NPM_TOKEN` 需要有发布 `@kunlunxu/brick` 包的权限
- 默认通过 `GITHUB_TOKEN` 推送版本提交、tag 和创建 GitHub Release，无需额外配置

## 发版流程

项目发版由 `.github/workflows/release.yml` 自动完成，主要流程如下：

1. 生成版本提交、`CHANGELOG.md` 和 git tag
2. 推送版本提交和 tag
3. 创建 GitHub Release
4. 编译 npm 包产物
5. 发布 npm 包

### 自动发版

当代码合并并 `push` 到 `master` 分支后，会自动触发 `Release workflow`。

```shell
git checkout master
git pull
git push origin master
```

`workflow` 会执行：

```shell
npm ci
npm run release
git push --follow-tags
npm run build:publish
npm publish --access public
```

发布完成后会生成:

- `chore(release): ...` 版本提交
- `vX.Y.Z` tag
- `GitHub Release`
- `npm` 包新版本

### 补充: 版本号递增规则

自动发版会执行：

```shell
npm run release
```

该命令会根据提交信息自动判断版本号递增方式：

- `fix:` 对应 patch 版本，例如 `2.6.4` -> `2.6.5`
- `feat:` 对应 minor 版本，例如 `2.6.4` -> `2.7.0`
- `BREAKING CHANGE`、`feat!:` 或 `fix!:` 对应 major 版本，例如 `2.6.4` -> `3.0.0`

提交信息建议遵循 Conventional Commits：

```shell
fix: 修复 markdown 样式问题
feat: 新增 editor onCopy 配置
feat!: 调整组件导出结构
```

如果需要跳过自动判断并指定版本号，使用手动指定版本发版。

### 手动指定版本发版

如果需要指定版本号，可以在 GitHub Actions 页面手动触发：

```text
Actions -> Release -> Run workflow
```

在 `release_as` 中填写目标版本号，例如：

```text
2.6.5
```

对应本地命令等价于：

```shell
npm run release -- --release-as 2.6.5
```

### 本地手动发版

正常情况下优先使用 GitHub Actions 发版。只有当 CI 不可用，或需要本地排查发版问题时，再使用本地手动流程。

- 拉取远程仓库数据

```shell
# 拉取远程数据包括 tag 信息
git fetch origin --prune
```

- 版本发布， 生成 changelog 和  tag

```shell
# 自动根据 commit 生成版本
npm run release

# 或指定版本号
npm run release -- --release-as 2.6.5
```

```shell
# 如遇如下问题请删除 v1.1.5 分支
✔ bumping version in package.json from 1.1.6 to 1.1.6
Error in git-raw-commits: warning: refname 'v1.1.5' is ambiguous.

```

- push commit 和 tag 到远程

```shell
git push --follow-tags
```

- 创建 github releases: 直接讲 changelog 当前版本信息拷贝过去

- npm 包编译

```shell
npm run build:publish
```

- 发布包

```shell
# 1. 切换官方源
npm config set registry https://registry.npmjs.org

# 2. 登录 npm
npm login

# 3. 发布包
npm publish --access public

# 4. 如果需要则切换回淘宝源 / https://registry.yarnpkg.com
npm config set registry https://registry.npmmirror.com
```

### 发版注意事项

- `chore(release):` 提交会被 Release workflow 跳过，避免发版提交再次触发发版
- 如果 tag 已存在，需要先确认 npm 和 GitHub Release 是否已经发布成功，不要直接重复发布
- 如果 `npm publish` 失败，先确认 `NPM_TOKEN` 是否有效，以及当前版本是否已经存在于 npm
- 如果 GitHub Release 创建失败，可以根据 `CHANGELOG.md` 中当前版本内容手动补建 Release

```shell
npm view @kunlunxu/brick version
```
